Section 01

Installing Git

Git is the version control system powering our entire workflow. Install it on your machine before anything else.

🪟 Windows

PowerShell / Winget
# Option 1 — Using winget (recommended)
winget install Git.Git

# Option 2 — Download installer from
# https://git-scm.com/download/win
# Run installer → keep all defaults → select "Git Bash Here"

🍎 macOS

Terminal
# Option 1 — Homebrew (recommended)
brew install git

# Option 2 — Xcode Command Line Tools
xcode-select --install

🐧 Linux (Ubuntu / Debian)

Bash
sudo apt update
sudo apt install git -y

✅ Verify Installation

Any Terminal
git --version
# Expected output: git version 2.xx.x
✅You need Git 2.28 or higher for all Team-B workflows to function correctly. Update if your version is older.
⚙️
Section 02

Initial Git Configuration

Before your first commit, configure Git with your identity. This information is permanently embedded in every commit you make.

⚠️Use the same email as your GitHub account. Mismatched emails break contribution graphs and CODEOWNERS attribution.
Terminal
# Set your display name
git config --global user.name "Your Full Name"

# Set your GitHub-registered email
git config --global user.email "you@example.com"

# Set default branch name to "main"
git config --global init.defaultBranch main

# Set VS Code as default editor (recommended)
git config --global core.editor "code --wait"

# Enable coloured output in terminal
git config --global color.ui auto

# Set merge strategy
git config --global pull.rebase false

# Verify all settings
git config --global --list

Developer-Specific Examples

Section 03

SSH Key Setup

SSH keys allow you to authenticate with GitHub securely — no passwords required. Each developer must generate their own SSH key pair.

ℹ️An SSH key pair consists of a private key (stays on your machine, never share) and a public key (uploaded to GitHub).
1

Generate the SSH Key

Terminal
# Replace with your actual GitHub email
ssh-keygen -t ed25519 -C "you@example.com"

# When prompted for file location:
# Press Enter for default (~/.ssh/id_ed25519)
# Or type a custom name for multiple accounts:
# e.g., /Users/you/.ssh/id_teamb

# Set a passphrase (recommended for security)
2

Start the SSH Agent

Terminal
# Start the agent in background
eval "$(ssh-agent -s)"

# Add your private key to agent
ssh-add ~/.ssh/id_ed25519

# Windows (Git Bash): same commands above
# Windows (PowerShell):
Start-Service ssh-agent
ssh-add ~\.ssh\id_ed25519
3

Copy Your Public Key

Terminal
# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# Windows (PowerShell)
Get-Content ~\.ssh\id_ed25519.pub | Set-Clipboard

# Or just print and manually copy
cat ~/.ssh/id_ed25519.pub
🐙
Section 04

Adding SSH Key to GitHub

Once you have your public key copied, add it to your GitHub account so GitHub can authenticate you.

1

Open GitHub SSH Settings

Go to GitHub.com → Profile photo (top-right) → Settings → SSH and GPG keys → New SSH key

2

Fill in the Details

• Title: Use a descriptive name like MacBook-Work-2025 or Ubuntu-Dev
• Key type: Authentication Key
• Key: Paste your public key

3

Test the Connection

Terminal
ssh -T git@github.com

# Success output:
Hi username! You've successfully authenticated...

# If you see "permission denied", check key was added correctly
👥
Section 05

Using Multiple GitHub Accounts

If you have a personal GitHub account and your Team-B work account, you need separate SSH keys and an SSH config to manage both seamlessly.

⚠️Never reuse the same SSH key across multiple GitHub accounts. GitHub requires unique keys per account.

Generate a Second Key for Work

Terminal
# Personal key (already exists)
# ~/.ssh/id_ed25519

# Generate a separate key for Team-B work account
ssh-keygen -t ed25519 -C "you@teamb-company.com" -f ~/.ssh/id_teamb

# Add work key to SSH agent
ssh-add ~/.ssh/id_teamb

# Copy the work public key
cat ~/.ssh/id_teamb.pub

Then add id_teamb.pub to your Team-B GitHub account following the same steps in Section 04.

📄
Section 06

SSH Config File for Multiple Accounts

The SSH config file at ~/.ssh/config tells your system which key to use for which host. This is what makes multiple accounts work transparently.

~/.ssh/config
# ── Personal GitHub Account ──
Host github.com-personal
  HostName     github.com
  User         git
  IdentityFile ~/.ssh/id_ed25519

# ── Team-B Work Account ──
Host github.com-teamb
  HostName     github.com
  User         git
  IdentityFile ~/.ssh/id_teamb

Using the Config Aliases

When cloning, replace github.com with your configured host alias:

Terminal
# Clone personal project
git clone git@github.com-personal:your-username/personal-repo.git

# Clone Team-B project
git clone git@github.com-teamb:Team-B/repository.git

# Test each alias independently
ssh -T git@github.com-personal
ssh -T git@github.com-teamb
ℹ️You can also set per-repo Git configs with git config user.email "work@email.com" (without --global) inside the repo folder to override global settings.

Clone the Team-B repository to your local machine. Each developer has a dedicated folder inside developers/. Never work outside your own folder — CODEOWNERS rules will reject it.

Terminal

# Navigate into the repo
cd repository

git remote -v

# origin  git@github.com-teamb:Team-B/repository.git (fetch)
# origin  git@github.com-teamb:Team-B/repository.git (push)

Section 08

Branch Workflow

Team-B follows a structured branching strategy. Never push directly to main. All changes go through feature branches and pull requests.

Branch Naming Convention

TypeFormatExample
featurefeature/dev-name/descriptionfeature/rohit/login-page
fixfix/dev-name/issue-descriptionfix/tanuja/navbar-overflow
hotfixhotfix/dev-name/critical-issuehotfix/ajay/payment-crash
docsdocs/dev-name/what-changeddocs/maninder/api-reference
chorechore/dev-name/taskchore/akshay/update-deps

Daily Development Flow

Terminal — Step by Step
# Step 1: Always start from latest main
git checkout main
git pull origin main

# Step 2: Create your feature branch
git checkout -b feature/yourname/your-feature

# Step 3: Work on your files, then stage
git add developers/yourname/  # Only add YOUR folder

# Step 4: Commit with a good message (see Section 09)
git commit -m "feat: add user authentication module"

# Step 5: Push to remote
git push origin feature/yourname/your-feature

# Step 6: Open a PR on GitHub (see Section 10)

Branch Protection Summary

🔒 main branch
Direct pushes blocked. All merges require a PR with at least 1 approval. CI checks must pass.
🛡️ projects/ folder
Protected via CODEOWNERS. Changes require review from the assigned project owner.
👤 developers/ folder
Each developer owns their subfolder. GitHub Actions enforce you only touch your own directory.
✅ CI Validation
GitHub Actions run on every PR to validate folder structure, naming conventions, and permissions.
✍️
Section 09

Commit Message Standards

Team-B follows the Conventional Commits specification. Good commit messages make code review faster, changelogs automatic, and history readable.

Format

Format
type(optional-scope): short description

[optional body — explain WHY, not what]

[optional footer — issue references, breaking changes]

Commit Types

TypeWhen to useExample
featNew feature or functionalityfeat: add product search filter
fixBug fixesfix: resolve null pointer on login
docsDocumentation changes onlydocs: update README setup guide
styleFormatting, spacing (no logic change)style: reformat button component
refactorCode restructuring (no feature/fix)refactor: simplify auth middleware
testAdding or updating teststest: add unit tests for cart
choreBuild process, dependency updateschore: update npm packages

Real Examples

Good Commits ✅
feat(auth): implement JWT refresh token rotation
fix(bajaj-electricals): correct invoice total calculation
docs: add SSH multi-account setup guide
refactor(api): extract validation into separate middleware
Bad Commits ❌ — Avoid These
fixed stuff
WIP
changes
asdfghjkl
update
💡Keep the first line under 72 characters. Use imperative mood: "add feature" not "added feature". If you need more context, add it in the commit body after a blank line.

Section 10

Creating Pull Requests

All code changes to main go through Pull Requests. Our repo has a PR template — fill it out completely for faster reviews.

1

Push Your Branch

Terminal
git push -u origin feature/yourname/your-feature
# The -u flag sets upstream tracking for future pushes
2

Open a PR on GitHub

GitHub will show a banner after pushing. Click "Compare & pull request" or go to the Pull Requests tab and click New pull request.

3

Complete the PR Template

PR Template Checklist
## Summary
What does this PR do? Why is it needed?

## Changes Made
- [ ] Feature / Bug fix / Docs / Refactor
- [ ] Affected files listed

## Testing
- [ ] Manually tested locally
- [ ] No console errors

## Checklist
- [ ] Branch named correctly (feature/name/description)
- [ ] Only modified files within my developer folder
- [ ] Commit messages follow Conventional Commits
- [ ] PR title follows: feat: / fix: / docs: etc.
4

Assign Reviewers & Labels

Assign at least 1 reviewer. GitHub Actions will automatically request reviews from CODEOWNERS for protected paths. Add relevant labels like feature, bugfix, or the client project name.

5

Address Review Comments

Terminal
# Make requested changes, then:
git add .
git commit -m "fix: address review feedback on validation"
git push
# PR auto-updates — no need to close and reopen
✅Once approved and all CI checks pass, the PR author or a maintainer will merge. Squash merge is the default strategy to keep main history clean.
🔄
Section 11

Updating Your Local Repository

The repository evolves daily. Always sync before starting new work to avoid conflicts.

Daily Sync Routine

Terminal
# 1. Switch to main
git checkout main

# 2. Fetch + merge latest changes
git pull origin main

# 3. Checkout your feature branch
git checkout feature/yourname/your-feature

# 4. Rebase on top of latest main (preferred)
git rebase main

# — OR — merge if you prefer
git merge main

Handling Merge Conflicts

Terminal
# 1. After conflict is reported, check which files
git status

# 2. Open conflicted files — look for conflict markers:
<<<<<<< HEAD
your local code
=======
incoming code
>>>>>>> main

# 3. Edit file to keep the correct code, remove markers

# 4. Stage the resolved file
git add <conflicted-file>

# 5. Continue rebase
git rebase --continue

# — OR — if merging:
git commit
💡VS Code has a built-in merge conflict editor. When conflicts arise, open VS Code and use the "Accept Current / Incoming / Both" buttons for a visual resolution experience.

Section 12

Troubleshooting Common Issues

❌ SSH Permission Denied

Fix
# Check if SSH agent has keys loaded
ssh-add -l

# If empty, add your key again
ssh-add ~/.ssh/id_teamb

# Test connection verbosely
ssh -vT git@github.com-teamb

# Verify correct key is being used
ssh -i ~/.ssh/id_teamb -T git@github.com

❌ Wrong Author on Commits

Fix
# Fix last commit author
git commit --amend --reset-author

# Fix last N commits (example: last 3)
git rebase -i HEAD~3
# Change "pick" to "edit" for commits to fix
# Then: git commit --amend --reset-author
# Then: git rebase --continue

❌ Accidentally Committed to main

Fix
# Move commit to a new branch (save your work)
git branch feature/yourname/save-my-work

# Reset main to before the accidental commit
git reset --hard HEAD~1

# Switch to the new branch and continue
git checkout feature/yourname/save-my-work

❌ Undo Last Commit (keep changes)

Fix
# Undo commit, keep files staged
git reset --soft HEAD~1

# Undo commit, keep files unstaged
git reset HEAD~1

# DISCARD changes completely (destructive!)
git reset --hard HEAD~1

❌ PR Blocked by GitHub Actions

ErrorCauseFix
Folder validation failedModified files outside your developer folderCheck git diff --name-only main and revert unauthorized files
Permission denied on projects/Committed to a project folder you don't ownRevert those specific files, open a PR to the correct owner
Branch name invalidBranch doesn't match naming patternRename: git branch -m old-name feature/yourname/desc
CODEOWNERS review requiredChanged a protected pathWait for or request review from the code owner

❌ Stash Uncommitted Work

Terminal
# Save current changes without committing
git stash push -m "WIP: half-done feature"

# List all stashes
git stash list

# Restore latest stash
git stash pop

# Restore specific stash
git stash apply stash@{2}
