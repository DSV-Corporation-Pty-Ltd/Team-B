Getting Started

Installing Git

Git is the version control system powering all Team-B collaboration. Install it before anything else — you need Git 2.28 or higher.

Windows

PowerShell
# Recommended — using winget
winget install Git.Git

# Or download from https://git-scm.com/download/win
# Keep all defaults; select "Git Bash Here"

macOS

Terminal
# Homebrew (recommended)
brew install git

# Or Xcode Command Line Tools
xcode-select --install

Linux (Ubuntu / Debian)

Bash
sudo apt update && sudo apt install git -y

Verify

Any terminal
git --version
# Expected: git version 2.xx.x
✅
You need Git 2.28+. Update with brew upgrade git (macOS) or git update-git-for-windows (Windows).

Getting Started

Initial Configuration

Before your first commit, tell Git who you are. This identity is permanently embedded in every commit you make.

⚠️
Use the same email registered on your GitHub account. Mismatched emails break contribution graphs and CODEOWNERS attribution.
Terminal
# Display name embedded in commits
git config --global user.name "Your Full Name"

# GitHub-registered email
git config --global user.email "you@example.com"

# Default branch name
git config --global init.defaultBranch main

# Editor (VS Code)
git config --global core.editor "code --wait"

# Coloured output
git config --global color.ui auto

# Pull strategy
git config --global pull.rebase false

# Verify all settings
git config --global --list

Getting Started

SSH Key Setup

SSH keys authenticate you with GitHub without passwords. The private key stays on your machine — never share it. The public key goes to GitHub.

1

Generate the Key Pair

Terminal
# Replace with your GitHub email
ssh-keygen -t ed25519 -C "you@example.com"

# File location: press Enter for default (~/.ssh/id_ed25519)
# Or name it:    ~/.ssh/id_teamb   (for multi-account)
# Passphrase:    recommended — set one
2

Start the SSH Agent & Add Your Key

macOS / Linux
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
Windows PowerShell
Start-Service ssh-agent
ssh-add ~\.ssh\id_ed25519
3

Copy Your Public Key

Terminal
# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# Windows
Get-Content ~\.ssh\id_ed25519.pub | Set-Clipboard

# Or print and copy manually
cat ~/.ssh/id_ed25519.pub

Getting Started

Adding SSH Key to GitHub

With your public key copied, register it on GitHub so the platform can verify your identity.

1

Open SSH Settings

GitHub.com → Profile photo → Settings → SSH and GPG keys → New SSH key

2

Fill the Form

Title: A recognisable label like MacBook-Work-2025
Key type: Authentication Key
Key: Paste your public key

3

Test the Connection

Terminal
ssh -T git@github.com

# Success looks like:
Hi username! You've successfully authenticated...

# "Permission denied"? → See Section 12

Multi-Account

Multiple GitHub Accounts

Have a personal GitHub account alongside your Team-B work account? You need a dedicated SSH key for each. GitHub requires unique keys per account.

⚠️
Never reuse the same SSH key across multiple GitHub accounts — GitHub will reject it on the second one.
Terminal
# Personal key already exists at: ~/.ssh/id_ed25519

# Generate a dedicated key for Team-B
ssh-keygen -t ed25519 -C "you@teamb.com" -f ~/.ssh/id_teamb

# Add work key to the agent
ssh-add ~/.ssh/id_teamb

# Print public key → add to your Team-B GitHub account
cat ~/.ssh/id_teamb.pub

Add id_teamb.pub to your work GitHub account following the same steps as Section 04.

06
Multi-Account

SSH Config File

The file at ~/.ssh/config maps host aliases to specific keys. This makes multiple accounts transparent in daily use.

~/.ssh/config
# ── Personal Account ─────────────────────
Host github.com-personal
  HostName     github.com
  User         git
  IdentityFile ~/.ssh/id_ed25519

# ── Team-B Work Account ──────────────────
Host github.com-teamb
  HostName     github.com
  User         git
  IdentityFile ~/.ssh/id_teamb

Using Your Aliases

Terminal
# Clone personal project
git clone git@github.com-personal:username/repo.git

# Clone Team-B project
git clone git@github.com-teamb:Team-B/repository.git

# Test each alias
ssh -T git@github.com-personal
ssh -T git@github.com-teamb
💡
Override identity per-repo: run git config user.email "work@email.com" without --global inside the repo folder.

Daily Workflow

Branch Workflow

Team-B uses a structured branching model. Never push directly to main — it is branch-protected. All changes go through pull requests.

Naming Convention

TypePatternExample
featurefeature/devname/descriptionfeature/rohit/login-page
fixfix/devname/descriptionfix/tanuja/navbar-overflow
hotfixhotfix/devname/descriptionhotfix/ajay/payment-crash
docsdocs/devname/descriptiondocs/maninder/api-reference
chorechore/devname/descriptionchore/akshay/update-deps

Daily Flow

Terminal — step by step
# 1. Always start from latest main
git checkout main
git pull origin main

# 2. Create your branch
git checkout -b feature/yourname/your-feature

# 3. Stage ONLY your folder
git add developers/yourname/

# 4. Commit (see Section 09 for message format)
git commit -m "feat: add authentication module"

# 5. Push
git push origin feature/yourname/your-feature

# 6. Open a PR on GitHub → see Section 10

Protection Summary

🔒 main branch
Direct pushes blocked. PRs require 1 approval + passing CI checks.
🛡️ projects/ folder
CODEOWNERS triggers automatic review from the project owner.
👤 developers/ folder
GitHub Actions enforce you only modify your own subfolder.
⚡ CI Validation
Actions validate folder structure, naming, and permissions on every PR.

Commit Message Standards

Team-B follows the Conventional Commits specification. Good messages make reviews faster, changelogs automatic, and history readable.

Format

Structure
type(optional-scope): short description — under 72 chars

[body: explain WHY, not what — optional]

[footer: issue refs, breaking changes — optional]

Type Reference

TypeUse forExample
featNew featurefeat: add product search filter
fixBug fixfix: resolve null pointer on login
docsDocumentation onlydocs: update README setup guide
styleFormatting, no logicstyle: reformat button component
refactorCode restructurerefactor: simplify auth middleware
testTeststest: add unit tests for cart
choreDeps, build configchore: update npm packages
Good examples ✅
feat(auth): implement JWT refresh token rotation
fix(bajaj-electricals): correct invoice total rounding
docs: add SSH multi-account setup guide
refactor(api): extract validation into middleware
Avoid ❌
fixed stuff    ← vague
WIP             ← incomplete
changes         ← meaningless
update          ← update what?
💡
Use imperative mood: "add feature" not "added feature". Keep the first line under 72 characters.
10
Daily Workflow

Creating Pull Requests

All changes to main go through pull requests. Our repo ships with a PR template — fill it out completely for faster reviews.

1

Push Your Branch

Terminal
git push -u origin feature/yourname/your-feature
# -u sets upstream tracking for future pushes
2

Open the PR

GitHub shows a yellow banner after pushing — click "Compare & pull request". Or go to Pull Requests → New pull request.

3

Complete the Template

PR template
## Summary
What does this PR do and why?

## Type of Change
- [ ] Feature  - [ ] Bug fix  - [ ] Docs  - [ ] Refactor

## Testing
- [ ] Tested locally
- [ ] No console errors

## Checklist
- [ ] Branch named correctly (type/devname/desc)
- [ ] Only modified my developer folder
- [ ] Commits follow Conventional Commits
- [ ] PR title uses type prefix: feat: / fix: / etc.
4

Assign Reviewers & Labels

Assign at least 1 reviewer. CODEOWNERS automation requests reviews for protected paths automatically. Add labels for the type and client project.

5

Address Review Comments

Terminal
git add .
git commit -m "fix: address review feedback"
git push
# PR auto-updates — no need to close and reopen
✅
Once approved and CI passes, the PR is merged with Squash merge to keep main history clean.

Daily Workflow

Sync & Update

The repository evolves daily across 10 developers. Always sync before starting new work.

Daily Sync

Terminal
# 1. Pull latest main
git checkout main
git pull origin main

# 2. Switch to your branch
git checkout feature/yourname/your-feature

# 3. Rebase onto latest main (keeps history clean)
git rebase main

# — or merge if preferred —
git merge main

Resolving Merge Conflicts

Terminal
# 1. Check conflicting files
git status

# 2. Find conflict markers in the file:
<<<<<<< HEAD
your local changes
=======
incoming changes
>>>>>>> main

# 3. Edit file — keep the right code, remove markers

# 4. Stage resolved file
git add <filename>

# 5. Continue rebase
git rebase --continue
💡
VS Code has a built-in merge editor — use the Accept Current / Incoming / Both buttons for a visual resolution experience.

Stashing Unfinished Work

Terminal
git stash push -m "WIP: half-done feature"
git stash list
git stash pop          # restore latest
git stash apply stash@{2}  # restore specific

Help

Troubleshooting

SSH: Permission Denied

Fix
# Check loaded keys
ssh-add -l

# Re-add if empty
ssh-add ~/.ssh/id_teamb

# Verbose test
ssh -vT git@github.com

Wrong Author on Commits

Fix
# Fix last commit
git commit --amend --reset-author

# Fix last 3 commits
git rebase -i HEAD~3
# Mark commits as "edit", then for each:
#   git commit --amend --reset-author
#   git rebase --continue

Accidentally Committed to main

Fix
# 1. Save the commit to a new branch
git branch feature/yourname/save-my-work

# 2. Reset main back
git reset --hard HEAD~1

# 3. Continue on the new branch
git checkout feature/yourname/save-my-work

Undo Last Commit (keep files)

Terminal
git reset --soft HEAD~1   # keep files staged
git reset HEAD~1          # keep files unstaged
git reset --hard HEAD~1   # ⚠ discard changes (destructive)
